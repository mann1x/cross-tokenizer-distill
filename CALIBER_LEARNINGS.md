- **[gotcha]** Ollama `/api/generate` caps `top_logprobs` at 20 â€” requesting 32 returns `{"error":"top_logprobs must be between 0 and 20"}`. Together / Fireworks / DeepInfra echo+logprobs share the same 20 cap. CTD's default K=32 is unreachable through any of these backends; build remote-teacher caches with Kâ‰¤20.
- **[gotcha]** Ollama cloud-tagged models (`:cloud` suffix, e.g. `qwen3-coder:480b-cloud`) silently strip the `logprobs` field from `/api/generate` responses even when `logprobs: true` is requested. The local backend returns logprobs; the cloud backend does not. Verify with `'logprobs' in response` before relying on cloud Ollama as a teacher source.
- **[gotcha]** Ollama `/api/generate` does NOT support echo / forced-decode â€” there's no way to get logprobs for an existing token sequence in one call. Each teacher position requires its own HTTP roundtrip with a growing prefix. Hosted OpenAI-compat APIs (Together / Fireworks / DeepInfra) DO support `echo: true` + `logprobs: K` for one-call-per-example precompute.
- **[gotcha]** Scripts under `experiments/validation/` aren't run as a package â€” importing `ctd.*` from there fails with `ModuleNotFoundError: No module named 'ctd'`. Set `PYTHONPATH=/srv/dev-disk-by-label-opt/dev/cross-tokenizer-distill` (repo root) before invoking, or `cd` to the repo root and run with the full subpath.
- **[env]** The project's working transformers env is `/shared/dev/lightseek/.venv` (transformers 5.5.0, Python 3.10) â€” system Python 3.9 lacks newer transformers and chokes on Gemma-4 tokenizer configs (`AttributeError: 'list' object has no attribute 'keys'`). `source /shared/dev/lightseek/.venv/bin/activate` before running anything that loads modern HF tokenizers.
- **[env]** Local Ollama on solidpc/pandorum is exposed at port `11434` directly and via the proxy at `solidpc:11433` (claude-hooks proxy). Port `pandorum:11433` is NOT routable â€” the proxy only listens on solidpc. Probe with `curl -sf http://solidpc:11433/api/version` first.
- **[gotcha]** `sqlite3 "SELECT printf(\"%.1f%%\", ...)"` from a shell heredoc/ssh quoting often mangles the `%` and percent-format string into `no such column: %.1f%%`. When pulling eval tallies, prefer plain `SELECT task, COUNT(*), SUM(passed)` and format in Python, or use single-quoted printf inside a properly-escaped sqlite3 call.
- **[gotcha]** Vast.ai pod base images don't ship `column` (bsdmainutils) â€” `column -t -s '\t'` returns `bash: column: command not found`. For TSV inspection over ssh, use `python3 -c` or `awk -F'\t'` instead.
- **[gotcha]** `ssh ... 'python3 -c "..."'` with embedded `\"` escapes for f-strings frequently returns empty stdout (silent quoting failure). Use a heredoc form `ssh ... "python3 << 'EOF' ... EOF"` with single-quoted EOF marker for any non-trivial Python over ssh.
- **[gotcha]** WebFetch on `github.com/.../blob/...` and the `raw.githubusercontent.com` equivalent both returned empty for this repo's docs â€” use the local clone (`/shared/dev/cross-tokenizer-distill/docs/`) directly when reading project documentation rather than fetching from the web mirror.
- **[gotcha]** In `adapters/transformers.py` `CTDTrainerMixin.compute_loss`, student logits are passed UNSHIFTED to the distill loss while HF CE shifts internally â€” the comment admits "Default: assume aligned" but `05_train.py` builds `ctd_positions = range(cache_offset, cache_offset + len(ids))` aligned to position `i`, so distill compares student logit `i` against the teacher distribution for predicting token `i` (not `i+1`). Verify shift convention before tuning recipe knobs; an off-by-one explains "distill degrades vs SFT" symptoms more readily than weight/temperature.
- **[gotcha]** The CTD precompute cache for run_C is named `qwen25_coder_7b_via_ctd_top32.pt` (note the `_via_` infix), NOT `qwen25_coder_7b_ctd_top32.pt`. Always `ls cache_C/` to confirm the exact filename before scripting `--cache` args â€” silent FileNotFoundError on the cache leaves an empty `runs/<run>/` dir, then the smoke step fails with the misleading `Can't find 'adapter_config.json'` from PEFT and the gate logic falls through (gate var empty â‰  "PASS" so script exits, but root cause is upstream).
- **[pattern]** When chaining trainâ†’smokeâ†’full-eval in a single bash script with `set -uo pipefail` (no `-e`), a missing-file failure in step 1 cascades silently through step 2 (smoke ran on empty adapter dir â†’ PEFT error â†’ no JSON â†’ gate var unset â†’ "FAIL"). Either add `set -e` after the cache existence check, or `ls -la $CACHE_C` as an explicit precondition + `[ -f $CACHE_C ] || exit 2` at script top so the failure is loud and immediate.
- **[gotcha]** `caliber init` flags are tightly restricted â€” `--auto-approve` causes the CLI to hang indefinitely (had to be SIGKILLed after 60s in foreground; backgrounding it just defers the hang). The supported `caliber refresh` flags are `--quiet` and `--dry-run` only; do NOT pass `--auto-approve`, `--force`, `--debug`, or any other invented flag. To install the pre-commit hook, use `caliber hooks --install` (not `caliber init`).
- **[pattern]** For same-vocab on-policy KL training in `06_train_onpolicy_xv_v2.py`, prefer the `IdentityMapper` + `_identity_alignment` fast path (auto-detected via `tokenizers_match(t_tok, s_tok)`) over `--mapper-cache`. Skips ~30s VocabMapper build, ~50MB cache file, decode/re-encode + UTF-8 offset alignment per sample. Smoke gate: log line `same-vocab detected ... using IdentityMapper, skipping mapper cache` + `mapper coverage: single=100.0%`.
- **[pattern]** When patching the CTD library against a remote vast.ai pod, sync via `scp` to `/tmp/ctd_patch_files/` then `cp` into both `/workspace/cross-tokenizer-distill/ctd/` AND `/workspace/cross-tokenizer-distill/experiments/validation/`. A direct `scp` to a non-existent destination dir fails silently (returns `{}` from Bash tool) â€” `mkdir -p` the staging dir on the pod first.
- **[gotcha]** `06_eval_batched.py` takes `--adapter` (NOT `--lora-adapter`); using the wrong flag yields `unrecognized arguments` but only AFTER training completes, wasting the train cycle. Sequential trainer scripts (`06_train_sft_on_teacher.py`) save final adapter at `<output_dir>/` plus per-epoch dirs `epoch-1/`, `epoch-2/`, NOT HF-style `checkpoint-*/`. Adapter-discovery helpers must check both `epoch-*` and `checkpoint-*` glob patterns and fall back to `$OUT` itself.
- **[pattern]** Multi-recipe driver scripts (train+eval pairs in a loop) MUST be idempotent: skip if `$JSON` exists, skip-train + eval-only if adapter dir is populated. Otherwise a fix-and-relaunch after one bad eval flag re-trains every previously-completed recipe from scratch. Pattern: `[ -f $JSON ] && return; [ -d $(adapter_path $OUT) ] && [ ... != $OUT ] && skip-train`.
- **[pattern]** When killing a multi-recipe driver mid-run to fix a bug, `kill -TERM <driver-pid>` does NOT propagate to the active python child (orphaned to PID 1, keeps running on GPU). To preserve in-flight work: `kill -TERM` only the driver bash, let the current python child finish, then relaunch the (now idempotent) driver â€” it'll resume at the next recipe.
- **[gotcha]** `06_eval_batched.py` chat-mode extractor (`extract_code_chat`) must use `FENCE_RE_REGEX.findall(t)[-1]`, NOT `.search(t)` â€” chat models often emit explanatory snippets in the FIRST fence and the actual answer in the LAST one. The chat path also needs `truncate_at_function_end()` applied when the extracted code starts with `def`/`class` (otherwise trailing prose after the function crashes `exec()`). Both `score_humaneval` and `score_mbpp` chat branches need this â€” verified via consultants audit csl-...-bd96.

- **[gotcha]** Chat-mode teacher completions from instruct models (Qwen2.5-Coder-14B-Instruct etc.) are typically 55-60% prose / 40-45% code by character count, even on bare "Write a function to ..." prompts. SFT loss masks that weight every continuation token equally pull the student toward verbose chat behavior and degrade pass@1 (v2 M38 MBPP regressed 22.5pp vs v1). Either constrain teacher output via a `--system-prompt` ("output ONLY the implementation, wrap in a single ```python block and stop") at regen time, or use `--code-only-mask` in `06_train_sft_on_teacher.py` to restrict loss to tokens inside ```python ... ``` fences via `return_offsets_mapping=True`. Audit any new chat-mode corpus's prose-share before training.
- **[env]** `claude-consultants` binary lives at `/root/.local/bin/claude-consultants` â€” it's NOT on `$PATH` for non-interactive bash and `which claude-consultants` returns nothing. Always invoke via the full path. CLI uses subcommands `config show` / `config set-effort <tier>` / `consult --cwd <dir> --message <text>` / `status <sid>` / `result <sid>` (returns JSON with `summary_markdown` + `metadata`). Follow-ups use `claude-consultants follow-up <parent-sid> --cwd <dir> --message <text>` â€” do NOT pass `<sid>` of an in-flight session, and a `close` on a still-running session returns HTTP 409.
- **[gotcha]** `claude-consultants` synthesizer occasionally hits Ollama HTTP 500 (`Internal Server Error`) mid-session and the session ends in `status=failed` with the researcher/critic outputs intact but no `summary_markdown`. Re-issue the same `follow-up` with the same body â€” the council's prior context is preserved via `--parent-sid` (or implicit when following-up the original parent sid).
- **[gotcha]** `--code-only-mask` is corpus-dependent: helps code-dense data (funcsig: M41c +3.0 HE / +2.4 MBPP vs M41) but hurts prose-heavy NLâ†’code data on same-vocab students (mbpp_train: M40c -2.4 HE / -2.1 MBPP vs M40). On cross-vocab the mask is robust across both corpora (M38c +6.1 HE on mbpp_train). Audit prose-share + decide mask use per (vocab-relation, corpus) pair, not globally.
- **[gotcha]** LoRA rank capacity defeats `--code-only-mask` regularization: at r=16 the mask inverts QC-1.5B HE-drain (M41c HE +0.6 vs base), at r=64 the same recipe collapses back to no-mask HE (M41bc HE -3.0). Capacity is an MBPP lever (M41bc MBPP +5.3, the best); mask cleanliness is an HE lever â€” pick rank by which metric you're optimizing.
