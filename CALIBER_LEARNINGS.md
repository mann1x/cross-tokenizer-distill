To install the pre-commit hook, use `caliber hooks --install` (not `caliber init`).

- **[gotcha]** `06_eval_batched.py` takes `--adapter` (NOT `--lora-adapter`); using the wrong flag yields `unrecognized arguments` AFTER training completes, wasting the train cycle. Sequential trainer scripts (`06_train_sft_on_teacher.py`) save final adapter at `<output_dir>/` plus per-epoch dirs `epoch-1/`, `epoch-2/`, NOT HF-style `checkpoint-*/`. - **[pattern]** When killing a multi-recipe driver mid-run to fix a bug, `kill -TERM <driver-pid>` does NOT propagate to the active python child (orphaned to PID 1). To preserve in-flight work: `kill -TERM` only the driver bash, let the current python child finish, then relaunch the (now idempotent) driver — it'll resume at the next recipe.
- **[gotcha]** `06_eval_batched.py` chat-mode extractor (`extract_code_chat`) must use `FENCE_RE_REGEX.findall(t)[-1]`, NOT `.search(t)` — chat models often emit explanatory snippets in the FIRST fence and the actual answer in the LAST one. The chat path also needs `truncate_at_function_end()` applied when the extracted code starts with `def`/`class`.
- **[env]** `claude-consultants` binary lives at `/root/.local/bin/claude-consultants` — not on `$PATH` for non-interactive bash. Always invoke via the full path. CLI uses subcommands `config show` / `config set-effort <tier>` / `consult --cwd <dir> --message <text>` / `status <sid>` / `result <sid>`. Follow-ups use `claude-consultants follow-up <parent-sid> --cwd <dir> --message <text>` — do NOT pass `<sid>` of an in-flight session, and `close` on a still-running session returns HTTP 409.
- **[gotcha]** `ssh ... 'python3 -c "..."'` with embedded `\"` escapes for f-strings frequently returns empty stdout (silent quoting failure) — and f-string expressions cannot include backslashes at all (`SyntaxError: f-string expression part cannot include a backslash`). Use a heredoc form `ssh ... "python3 << 'EOF' ... EOF"` with single-quoted EOF marker for any non-trivial Python over ssh, and avoid `\"` inside f-string `.get(\"key\")` — assign to a local var first. The same trap bites bash command-substitution: an inline regex like `r'```(?:python)?\s*\n...```'` inside `python3 -c "..."` makes bash try to execute `(?:python)?\s*\n` as a process substitution — ALWAYS write the audit script to a `.py` file and `scp` it.
- **[gotcha]** Chat-mode teacher completions from instruct models (Qwen2.5-Coder-14B-Instruct etc.) are typically 55-60% prose / 40-45% code by character count. SFT loss masks that weight every continuation token equally pull the student toward verbose chat behavior and degrade pass@1. Either constrain teacher output via `--system-prompt` ("output ONLY the implementation, wrap in a single ```python block and stop") at regen time, or use `--code-only-mask` in `06_train_sft_on_teacher.py` to restrict loss to tokens inside ```python ... ``` fences.
- **[gotcha]** `claude-consultants` synthesizer occasionally hits Ollama HTTP 500 mid-session and the session ends in `status=failed` with researcher/critic outputs intact but no `summary_markdown`. Re-issue the same `follow-up` — the council's prior context is preserved.
- **[gotcha]** `--code-only-mask` is corpus-dependent: helps code-dense data (funcsig: M41c +3.0 HE / +2.4 MBPP) but hurts prose-heavy NL→code data on same-vocab students (mbpp_train: M40c -2.4 HE / -2.1 MBPP). On cross-vocab the mask is robust across both corpora. Audit prose-share + decide mask use per (vocab-relation, corpus) pair, not globally.
- **[gotcha]** LoRA rank capacity defeats `--code-only-mask` regularization: at r=16 the mask inverts QC-1.5B HE-drain, at r=64 the same recipe collapses back to no-mask HE. Capacity is an MBPP lever; mask cleanliness is an HE lever — pick rank by which metric you're optimizing.

- **[gotcha]** NEVER `scp` to overwrite a bash script that an active `nohup`/background process is currently executing. Bash reads scripts lazily by file offset; rewriting bytes shifts the offset map and bash continues reading from the NEW content mid-byte, producing nonsensical errors like `ut: command not found`. To patch a long-running driver: write a SEPARATE recovery file or kill the process first.

- **[pattern]** Pod-side multi-step drivers should structure each step as a self-contained idempotent function (`[ -f $JSON ] && return` at top) and use a separate file per phase rather than one monolithic script. This makes mid-run patches safe and lets idle-barrier `while pgrep -f 06_train_sft|06_eval_batched; do sleep 30; done` sequence phases without coordination.
- **[gotcha]** A driver script that begins with `while pgrep -f "06_train_sft|06_eval_batched|..." > /dev/null; do sleep 30; done` (idle-barrier) self-deadlocks if its OWN train command matches the pgrep regex. Either narrow the barrier regex to exclude the about-to-launch process name, or skip the barrier when launching a single recipe interactively.
- **[pattern]** To launch a long-running python process over ssh that survives the ssh exit cleanly: write the command to a `.sh` file with a heredoc on the pod, then `nohup setsid /tmp/launch.sh > log 2>&1 < /dev/null & disown`. Plain `ssh ... 'nohup python ... &'` frequently returns empty stdout and the python child never actually starts. Always verify with a follow-up `pgrep -af` and `nvidia-smi`.
