# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** Wrap the per-problem generate+score loop in `try/except` for code-eval runners — a single problematic input that crashes generation (CUDA assert, tokenizer edge case) should become one recorded failure, not kill the entire 164-problem eval. Add `flush=True` to milestone prints so a stalled process is distinguishable from buffered stdout.
- **[pattern]** When a remote eval dies deterministically at the same step (e.g. HE 30/164 twice in a row) with no traceback in the log, the crash is almost certainly inside an unprotected `generate_one()` call — Python exceptions in `model.generate()` can terminate without flushing stderr to nohup logs. Add try/except + flush BEFORE relaunching, otherwise the third run dies the same way.
- **[pattern]** Use `setsid nohup bash script.sh < /dev/null > log 2>&1 &` (not `nohup ... & disown`) for remote pod relaunches — `disown` does not detach the process group, so SSH session SIGHUP can still reach the descendant. `setsid` puts it in a new session immune to terminal hangup.
- **[gotcha]** Code-eval `exec(prompt+gen+test)` MUST be wrapped in a per-problem `SIGALRM` timeout — a single generation containing `while True` or unbounded recursion hangs the entire eval at 100% CPU indefinitely (run_C HE/30 burned 71 min of wall-clock before being noticed). The process looks alive (state `R`, threads spinning) so pgrep monitors can't tell it from real progress. Default 30s mirrors EvalPlus.
- **[pattern]** Long pod evals must persist results per-problem to a SQLite cache (`PRAGMA journal_mode=WAL; synchronous=NORMAL`) the moment each is scored, NOT only as a final JSON dump. A SIGKILL or pod preemption mid-run otherwise loses every prior result; resumable cache makes kill-and-rerun a free operation. Skip already-cached `task_id`s on restart.
- **[gotcha]** A 0% pass@1 result on a code-eval task with non-trivial generations is almost always a SCORING bug, not a model failure. Probe one example end-to-end (decode → strip → truncate → exec) before assuming the model is broken.
- **[gotcha]** MBPP requires the standard prompt format (description + 1-3 assert tests embedded in a docstring) so the model knows the EXACT function name to define. A bare `prob['prompt']` ("Write a function to ...") makes the model hallucinate function names → 100% NameError at scoring even though generations are syntactically fine.
- **[pattern]** When monitoring a long pod job via `ssh ... 'nohup ... &'` from local Bash, the SSH client sometimes returns empty stdout despite the process launching successfully. Workaround: do `ssh ... 'cd X; nohup Y > log 2>&1 < /dev/null & disown; echo "launched pid $!"'` in a SINGLE ssh invocation, then verify with a second ssh that runs `sleep 8; pgrep -af` REMOTELY (not locally).
- **[pattern]** Long pod evals (3 adapters × HE+MBPP ≈ hours) outlive a single Monitor's 60-min timeout. Re-arm with progressively coarser filters as the run progresses (every-10 → every-20 → only DONE/pass@1 markers), and bump `sleep` from 90s → 150s → 180s to reduce ssh chatter.
- **[pattern]** Monitor commands watching remote long-running jobs MUST pair `tail | grep MILESTONES` with a `pgrep` for the process — emit `PROCESS-DIED` when pgrep is empty AND no DONE marker appears. A pure log-grep monitor goes silent forever when the process crashes.
- **[gotcha]** Code-eval `exec(prompt+gen+test)` MUST be wrapped in a per-problem `SIGALRM` timeout — a single generation containing `while True` or unbounded recursion hangs the entire eval at 100% CPU indefinitely. Default 30s mirrors EvalPlus.
- **[gotcha]** Ollama `/api/generate` caps `top_logprobs` at 20 — requesting 32 returns `{"error":"top_logprobs must be between 0 and 20"}`. Together / Fireworks / DeepInfra echo+logprobs share the same 20 cap. CTD's default K=32 is unreachable through any of these backends; build remote-teacher caches with K≤20.
- **[gotcha]** Ollama cloud-tagged models (`:cloud` suffix, e.g. `qwen3-coder:480b-cloud`) silently strip the `logprobs` field from `/api/generate` responses even when `logprobs: true` is requested. The local backend returns logprobs; the cloud backend does not. Verify with `'logprobs' in response` before relying on cloud Ollama as a teacher source.
- **[gotcha]** Ollama `/api/generate` does NOT support echo / forced-decode — there's no way to get logprobs for an existing token sequence in one call. Each teacher position requires its own HTTP roundtrip with a growing prefix. Hosted OpenAI-compat APIs (Together / Fireworks / DeepInfra) DO support `echo: true` + `logprobs: K` for one-call-per-example precompute.
- **[gotcha]** Scripts under `experiments/validation/` aren't run as a package — importing `ctd.*` from there fails with `ModuleNotFoundError: No module named 'ctd'`. Set `PYTHONPATH=/srv/dev-disk-by-label-opt/dev/cross-tokenizer-distill` (repo root) before invoking, or `cd` to the repo root and run with the full subpath.
- **[env]** The project's working transformers env is `/shared/dev/lightseek/.venv` (transformers 5.5.0, Python 3.10) — system Python 3.9 lacks newer transformers and chokes on Gemma-4 tokenizer configs (`AttributeError: 'list' object has no attribute 'keys'`). `source /shared/dev/lightseek/.venv/bin/activate` before running anything that loads modern HF tokenizers.
- **[env]** Local Ollama on solidpc/pandorum is exposed at port `11434` directly and via the proxy at `solidpc:11433` (claude-hooks proxy). Port `pandorum:11433` is NOT routable — the proxy only listens on solidpc. Probe with `curl -sf http://solidpc:11433/api/version` first.
- **[gotcha]** `sqlite3 "SELECT printf(\"%.1f%%\", ...)"` from a shell heredoc/ssh quoting often mangles the `%` and percent-format string into `no such column: %.1f%%`. When pulling eval tallies, prefer plain `SELECT task, COUNT(*), SUM(passed)` and format in Python, or use single-quoted printf inside a properly-escaped sqlite3 call.
- **[gotcha]** Vast.ai pod base images don't ship `column` (bsdmainutils) — `column -t -s '\t'` returns `bash: column: command not found`. For TSV inspection over ssh, use `python3 -c` or `awk -F'\t'` instead.
- **[gotcha]** `ssh ... 'python3 -c "..."'` with embedded `\"` escapes for f-strings frequently returns empty stdout (silent quoting failure). Use a heredoc form `ssh ... "python3 << 'EOF' ... EOF"` with single-quoted EOF marker for any non-trivial Python over ssh.
- **[gotcha]** WebFetch on `github.com/.../blob/...` and the `raw.githubusercontent.com` equivalent both returned empty for this repo's docs — use the local clone (`/shared/dev/cross-tokenizer-distill/docs/`) directly when reading project documentation rather than fetching from the web mirror.
- **[gotcha]** In `adapters/transformers.py` `CTDTrainerMixin.compute_loss`, student logits are passed UNSHIFTED to the distill loss while HF CE shifts internally — the comment admits "Default: assume aligned" but `05_train.py` builds `ctd_positions = range(cache_offset, cache_offset + len(ids))` aligned to position `i`, so distill compares student logit `i` against the teacher distribution for predicting token `i` (not `i+1`). Verify shift convention before tuning recipe knobs; an off-by-one explains "distill degrades vs SFT" symptoms more readily than weight/temperature.
- **[gotcha]** The CTD precompute cache for run_C is named `qwen25_coder_7b_via_ctd_top32.pt` (note the `_via_` infix), NOT `qwen25_coder_7b_ctd_top32.pt`. Always `ls cache_C/` to confirm the exact filename before scripting `--cache` args — silent FileNotFoundError on the cache leaves an empty `runs/<run>/` dir, then the smoke step fails with the misleading `Can't find 'adapter_config.json'` from PEFT and the gate logic falls through (gate var empty ≠ "PASS" so script exits, but root cause is upstream).
- **[pattern]** When chaining train→smoke→full-eval in a single bash script with `set -uo pipefail` (no `-e`), a missing-file failure in step 1 cascades silently through step 2 (smoke ran on empty adapter dir → PEFT error → no JSON → gate var unset → "FAIL"). Either add `set -e` after the cache existence check, or `ls -la $CACHE_C` as an explicit precondition + `[ -f $CACHE_C ] || exit 2` at script top so the failure is loud and immediate.
- **[gotcha]** `caliber init` flags are tightly restricted — `--auto-approve` causes the CLI to hang indefinitely (had to be SIGKILLed after 60s in foreground; backgrounding it just defers the hang). The supported `caliber refresh` flags are `--quiet` and `--dry-run` only; do NOT pass `--auto-approve`, `--force`, `--debug`, or any other invented flag. To install the pre-commit hook, use `caliber hooks --install` (not `caliber init`).
- **[pattern]** For same-vocab on-policy KL training in `06_train_onpolicy_xv_v2.py`, prefer the `IdentityMapper` + `_identity_alignment` fast path (auto-detected via `tokenizers_match(t_tok, s_tok)`) over `--mapper-cache`. Skips ~30s VocabMapper build, ~50MB cache file, decode/re-encode + UTF-8 offset alignment per sample. Smoke gate: log line `same-vocab detected ... using IdentityMapper, skipping mapper cache` + `mapper coverage: single=100.0%`.
- **[pattern]** When patching the CTD library against a remote vast.ai pod, sync via `scp` to `/tmp/ctd_patch_files/` then `cp` into both `/workspace/cross-tokenizer-distill/ctd/` AND `/workspace/cross-tokenizer-distill/experiments/validation/`. A direct `scp` to a non-existent destination dir fails silently (returns `{}` from Bash tool) — `mkdir -p` the staging dir on the pod first.
- **[gotcha]** `06_eval_batched.py` takes `--adapter` (NOT `--lora-adapter`); using the wrong flag yields `unrecognized arguments` but only AFTER training completes, wasting the train cycle. Sequential trainer scripts (`06_train_sft_on_teacher.py`) save final adapter at `<output_dir>/` plus per-epoch dirs `epoch-1/`, `epoch-2/`, NOT HF-style `checkpoint-*/`. Adapter-discovery helpers must check both `epoch-*` and `checkpoint-*` glob patterns and fall back to `$OUT` itself.
- **[pattern]** Multi-recipe driver scripts (train+eval pairs in a loop) MUST be idempotent: skip if `$JSON` exists, skip-train + eval-only if adapter dir is populated. Otherwise a fix-and-relaunch after one bad eval flag re-trains every previously-completed recipe from scratch. Pattern: `[ -f $JSON ] && return; [ -d $(adapter_path $OUT) ] && [ ... != $OUT ] && skip-train`.
- **[pattern]** When killing a multi-recipe driver mid-run to fix a bug, `kill -TERM <driver-pid>` does NOT propagate to the active python child (orphaned to PID 1, keeps running on GPU). To preserve in-flight work: `kill -TERM` only the driver bash, let the current python child finish, then relaunch the (now idempotent) driver — it'll resume at the next recipe.
