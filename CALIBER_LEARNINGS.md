# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** When chaining train→smoke→full-eval in a single bash script with `set -uo pipefail` (no `-e`), a missing-file failure in step 1 cascades silently through step 2 (smoke ran on empty adapter dir → PEFT error → no JSON → gate var unset → "FAIL"). Either add `set -e` after the cache existence check, or `ls -la $CACHE_C` as an explicit precondition + `[ -f $CACHE_C ] || exit 2` at script top so the failure is loud and immediate.
- **[gotcha]** `caliber init` flags are tightly restricted — `--auto-approve` causes the CLI to hang indefinitely (had to be SIGKILLed after 60s in foreground; backgrounding it just defers the hang). The supported `caliber refresh` flags are `--quiet` and `--dry-run` only; do NOT pass `--auto-approve`, `--force`, `--debug`, or any other invented flag. To install the pre-commit hook, use `caliber hooks --install` (not `caliber init`).
- **[pattern]** For same-vocab on-policy KL training in `06_train_onpolicy_xv_v2.py`, prefer the `IdentityMapper` + `_identity_alignment` fast path (auto-detected via `tokenizers_match(t_tok, s_tok)`) over `--mapper-cache`. Skips ~30s VocabMapper build, ~50MB cache file, decode/re-encode + UTF-8 offset alignment per sample. Smoke gate: log line `same-vocab detected ... using IdentityMapper, skipping mapper cache` + `mapper coverage: single=100.0%`.
- **[pattern]** When patching the CTD library against a remote vast.ai pod, sync via `scp` to `/tmp/ctd_patch_files/` then `cp` into both `/workspace/cross-tokenizer-distill/ctd/` AND `/workspace/cross-tokenizer-distill/experiments/validation/`. A direct `scp` to a non-existent destination dir fails silently (returns `{}` from Bash tool) — `mkdir -p` the staging dir on the pod first.
- **[gotcha]** `06_eval_batched.py` takes `--adapter` (NOT `--lora-adapter`); using the wrong flag yields `unrecognized arguments` but only AFTER training completes, wasting the train cycle. Sequential trainer scripts (`06_train_sft_on_teacher.py`) save final adapter at `<output_dir>/` plus per-epoch dirs `epoch-1/`, `epoch-2/`, NOT HF-style `checkpoint-*/`. Adapter-discovery helpers must check both `epoch-*` and `checkpoint-*` glob patterns and fall back to `$OUT` itself.
- **[pattern]** Multi-recipe driver scripts (train+eval pairs in a loop) MUST be idempotent: skip if `$JSON` exists, skip-train + eval-only if adapter dir is populated. Otherwise a fix-and-relaunch after one bad eval flag re-trains every previously-completed recipe from scratch. Pattern: `[ -f $JSON ] && return; [ -d $(adapter_path $OUT) ] && [ ... != $OUT ] && skip-train`.
- **[pattern]** When killing a multi-recipe driver mid-run to fix a bug, `kill -TERM <driver-pid>` does NOT propagate to the active python child (orphaned to PID 1, keeps running on GPU). To preserve in-flight work: `kill -TERM` only the driver bash, let the current python child finish, then relaunch the (now idempotent) driver — it'll resume at the next recipe.
- **[gotcha]** `06_eval_batched.py` chat-mode extractor (`extract_code_chat`) must use `FENCE_RE_REGEX.findall(t)[-1]`, NOT `.search(t)` — chat models often emit explanatory snippets in the FIRST fence and the actual answer in the LAST one. The chat path also needs `truncate_at_function_end()` applied when the extracted code starts with `def`/`class` (otherwise trailing prose after the function crashes `exec()`). Both `score_humaneval` and `score_mbpp` chat branches need this — verified via consultants audit csl-...-bd96.
- **[env]** `claude-consultants` binary lives at `/root/.local/bin/claude-consultants` — it's NOT on `$PATH` for non-interactive bash and `which claude-consultants` returns nothing. Always invoke via the full path. CLI uses subcommands `config show` / `config set-effort <tier>` / `consult --cwd <dir> --message <text>` / `status <sid>` / `result <sid>` (returns JSON with `summary_markdown` + `metadata`).
- **[gotcha]** `ssh ... 'python3 -c "..."'` with embedded `\"` escapes for f-strings frequently returns empty stdout (silent quoting failure). Use a heredoc form `ssh ... "python3 << 'EOF' ... EOF"` with single-quoted EOF marker for any non-trivial Python over ssh. The same trap bites bash command-substitution: an inline regex like `r'```(?:python)?\s*\n...```'` inside `python3 -c "..."` makes bash try to execute `(?:python)?\s*\n` as a process substitution — ALWAYS write the audit script to a `.py` file and `scp` it, never inline it.
- **[gotcha]** Chat-mode teacher completions from instruct models (Qwen2.5-Coder-14B-Instruct etc.) are typically 55-60% prose / 40-45% code by character count, even on bare "Write a function to ..." prompts. SFT loss masks that weight every continuation token equally pull the student toward verbose chat behavior and degrade pass@1 (v2 M38 MBPP regressed 22.5pp vs v1). Either constrain teacher output via a `--system-prompt` ("output ONLY the implementation, wrap in a single ```python block and stop") at regen time, or use `--code-only-mask` in `06_train_sft_on_teacher.py` to restrict loss to tokens inside ```python ... ``` fences via `return_offsets_mapping=True`. Audit any new chat-mode corpus's prose-share before training.
- **[env]** `claude-consultants` binary lives at `/root/.local/bin/claude-consultants` — it's NOT on `$PATH` for non-interactive bash and `which claude-consultants` returns nothing. Always invoke via the full path. CLI uses subcommands `config show` / `config set-effort <tier>` / `consult --cwd <dir> --message <text>` / `status <sid>` / `result <sid>` (returns JSON with `summary_markdown` + `metadata`). Follow-ups use `claude-consultants follow-up <parent-sid> --cwd <dir> --message <text>` — do NOT pass `<sid>` of an in-flight session, and a `close` on a still-running session returns HTTP 409.
- **[gotcha]** `claude-consultants` synthesizer occasionally hits Ollama HTTP 500 (`Internal Server Error`) mid-session and the session ends in `status=failed` with the researcher/critic outputs intact but no `summary_markdown`. Re-issue the same `follow-up` with the same body — the council's prior context is preserved via `--parent-sid` (or implicit when following-up the original parent sid).
- **[gotcha]** `--code-only-mask` is corpus-dependent: helps code-dense data (funcsig: M41c +3.0 HE / +2.4 MBPP vs M41) but hurts prose-heavy NL→code data on same-vocab students (mbpp_train: M40c -2.4 HE / -2.1 MBPP vs M40). On cross-vocab the mask is robust across both corpora (M38c +6.1 HE on mbpp_train). Audit prose-share + decide mask use per (vocab-relation, corpus) pair, not globally.
- **[gotcha]** LoRA rank capacity defeats `--code-only-mask` regularization: at r=16 the mask inverts QC-1.5B HE-drain (M41c HE +0.6 vs base), at r=64 the same recipe collapses back to no-mask HE (M41bc HE -3.0). Capacity is an MBPP lever (M41bc MBPP +5.3, the best); mask cleanliness is an HE lever — pick rank by which metric you're optimizing.
- **[convention]** `experiments/validation/runs/` (LoRA adapter dirs) is several GB and must stay out of git — add to `.gitignore` along with `.claude-hooks/` and `.claude/settings.local.json`. Eval result JSONs in `results/` are already gitignored.
- **[convention]** `experiments/validation/06_train_sft_on_teacher.py` saves per-epoch adapters at `<output-dir>/epoch-{n}/` (1-indexed) and a final copy at `<output-dir>/`. Adapter-discovery helpers should `ls -d $OUT/epoch-* | sort -V | tail -1` then fall back to `$OUT/checkpoint-*` then to `$OUT` itself.
- **[pattern]** When loading DS-Coder-V2 (or any HF checkpoint with custom modeling code in its repo), pass `trust_remote_code=True` to BOTH `AutoTokenizer.from_pretrained` AND `AutoModelForCausalLM.from_pretrained`. Safe to default ON: DS-Coder-1.3B and Qwen2.5-Coder ignore the flag. Pre-fix trainer scripts before launching V2-Lite runs — discovery-time crash mid-driver wastes the queue slot.
- **[gotcha]** NEVER `scp` to overwrite a bash script that an active `nohup`/background process is currently executing. Bash reads scripts lazily by file offset; rewriting bytes shifts the offset map and bash continues reading from the NEW content mid-byte, producing nonsensical errors like `ut: command not found`, `DB: unbound variable`, `syntax error near unexpected token `}'`. To patch a long-running driver: write a SEPARATE recovery file (e.g. `/tmp/run_v2_phase3c_recovery.sh`) and let the in-flight one finish, or kill the process first then overwrite. Idempotent drivers (skip-if-JSON-exists) make recovery trivial.
- **[gotcha]** Teacher-completion JSONLs from `gen_teacher_completions.py` only contain `{prompt, teacher_completion, prompt_token_len}` — the source `task_id` is dropped. Filtering teacher completions by `task_id` always yields 0 matches. To intersect with eval-result task_ids (e.g. for hard-prompt mining), build a `{task_id → prompt}` map from the original `*_prompts.jsonl` first, convert task_id sets to prompt-text sets, then filter teacher records by `rec['prompt'] in fail_prompts`.
- **[pattern]** Pod-side multi-step drivers should structure each step as a self-contained idempotent function (`[ -f $JSON ] && return` at top) and use a separate file per phase (`run_v2_phase3.sh`, `run_v2_phase3b_m45.sh`, `run_v2_phase3c_recovery.sh`) rather than one monolithic script. This makes mid-run patches safe (no overwrite of an active script) and lets idle-barrier `while pgrep -f 06_train_sft|06_eval_batched; do sleep 30; done` sequence phases without coordination.
- **[gotcha]** A driver script that begins with `while pgrep -f "06_train_sft|06_eval_batched|..." > /dev/null; do sleep 30; done` (idle-barrier) self-deadlocks if its OWN train command matches the pgrep regex — the barrier sees itself and waits forever. Either narrow the barrier regex to exclude the about-to-launch process name, or skip the barrier entirely when launching a single recipe interactively. Symptom: driver process visible in `pgrep` but `nvidia-smi` stays at `0 %, 1 MiB` indefinitely; bypass by `exec`ing the python directly via `nohup setsid ... < /dev/null > log 2>&1 & disown`.
- **[pattern]** To launch a long-running python process over ssh that survives the ssh exit cleanly, the reliable form is: write the command to a `.sh` file with a heredoc on the pod, then `nohup setsid /tmp/launch.sh > log 2>&1 < /dev/null & disown`. Plain `ssh ... 'nohup python ... &'` and even `ssh ... 'cd ... && nohup python ... > log 2>&1 &'` frequently return empty stdout (Bash tool result `{}`) and the python child never actually starts — confirmed multiple times this session. Always verify with a follow-up `pgrep -af` and `nvidia-smi` showing GPU memory > base.
- **[env]** wandb is NOT installed on vast.ai pods by default (`ModuleNotFoundError: No module named 'wandb'`, `wandb: command not found`). To enable wandb logging from a trainer: `pip install -q wandb` in the pod venv, then auth with `WANDB_API_KEY=<key> python3 -c 'import wandb; wandb.login(key="<key>")'` to write `~/.netrc`. The user's wandb credentials live in `/root/.config/wandb/settings` + `/root/.netrc` on solidPC (look for `machine api.wandb.ai` block). User is `mannix`; default project for this repo is `cross-tokenizer-distill`.
- **[gotcha]** Vast.ai pod overlay disk fills up fast (200G total). HF hub cache at `~/.cache/huggingface/hub/` is the largest consumer — by Phase 3 it held 94G across 6 model repos. Before launching a new large-model run (DSC-V2-Lite = 30G, QC-14B = 28G), `du -sh ~/.cache/huggingface/hub/* | sort -h` and `rm -rf` repos that aren't actively used (smaller students/teachers from prior phases). Always `df -h /workspace` before assuming download will succeed.
- **[correction]** Don't reinvent eval scripts — `06_eval_batched.py` already uses per-problem SQLite (`results/full_sft/*.db`) with `PRAGMA journal_mode=WAL`. Live progress is queryable mid-run via `sqlite3 <db> "SELECT task, COUNT(*), SUM(passed), ROUND(100.0*SUM(passed)/COUNT(*),1) FROM results GROUP BY task"`. The output JSON is only written at end-of-task — do NOT poll for the JSON to gauge progress, query the .db instead. Killing the process mid-eval is safe because completed rows are committed.
- **[pattern]** Default `06_eval_batched.py` loader builds with neither FA2 nor LoRA-merge, which on DS-Coder-V2-Lite (16B MoE) yields ~26s/problem at GPU 21% util and bs=8 — full HE+MBPP takes 4+ hours. For MoE models (and any LoRA eval) patch the loader to add `attn_implementation="flash_attention_2"` to both NF4 and BF16 `from_pretrained` paths AND `model = model.merge_and_unload()` after `PeftModel.from_pretrained` — combined with `--batch-size 16` this gives the 3-5x throughput the file's docstring claims. Mythic-RDT also ships a MoE-Vec patched `modeling_deepseek.py` at `/workspace/mythic-rdt/base/DeepSeek-Coder-V2-Lite-Instruct/` — point `--base-model` there instead of the HF hub repo to inherit it.
- **[gotcha]** `model = PeftModel.from_pretrained(base, args.adapter); model.merge_and_unload()` returns the merged model — must reassign `model = model.merge_and_unload()`, not call as a statement. After merge, the LoRA path is gone and inference matches a fully-finetuned checkpoint's speed.
- **[correction]** Default `06_eval_batched.py` loader (no FA2, no LoRA-merge) on DS-Coder-V2-Lite (16B MoE) gives ~26s/problem at 21% GPU util — that is the UNOPTIMIZED path. The optimized stack the user expects has THREE pieces wired in: (1) `attn_implementation="flash_attention_2"` on both NF4 + BF16 `from_pretrained`; (2) `model = model.merge_and_unload()` (must reassign — calling as a statement is a no-op) right after `PeftModel.from_pretrained`; (3) point `--base-model` at `/workspace/mythic-rdt/base/DeepSeek-Coder-V2-Lite-Instruct/` to inherit the MoE-Vec patched `modeling_deepseek.py`. Combined with `--batch-size 16` this drops gen time from ~26s → ~5.5s/problem (≥4x). Always patch the loader BEFORE launching MoE evals — mid-run kill+resume is safe (SQLite cache preserves completed rows), but wastes hours.
- **[convention]** `06_eval_batched.py` is resumable by design — kill the process and relaunch with the same `--cache-db` to skip already-completed rows. Use this freely when you discover an optimization (FA2, merge_and_unload, batch-size bump) mid-run rather than letting a slow eval finish.
